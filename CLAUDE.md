# LLM quickstart guide for chatgraph

This document orients an LLM assistant (or human reader) to the
chatgraph project. For the end-user view, see `README.md`; for
domain-specific schema walkthroughs, see `docs/` (e.g.
`docs/medical-schema.md` for the medical domain).

## What is chatgraph?

A voice-driven knowledge-elicitation demo. A subject speaks in their
own words about some topic in a chosen *domain*; an LLM-driven
assistant conducts an interview in real time; a second LLM extracts a
typed property graph of what was said, vertex by vertex and edge by
edge, into a live TinkerPop Gremlin Server. The first shipped domain
(`medical`) interviews a patient about headache disorders; the
architecture is domain-neutral.

## Where code lives

The repo follows the `src/main/<language>/<package>` /
`src/test/<language>/<package>` layout used across Hydra-family
projects (Hydra, HydraPop, the Hydra Python dist packages).

- `src/main/python/chatgraph/chat/` — voice loop, agent, extractor,
  graph writer, coordinator/CLI. The asyncio orchestration core.
- `src/main/python/chatgraph/domains/<name>/` — one self-contained
  subpackage per interview domain (schema builder, agent prompt,
  extractor prompt, opening line). `medical` is the only shipped
  domain so far.
- `src/main/python/chatgraph/schema/build.py` — CLI dispatcher
  (`chatgraph-build-schema <domain>`) that regenerates committed
  schema JSON from a domain's `schema_build.py`.
- `src/test/python/chatgraph/` — smoke tests. Some call real Claude
  APIs and cost money per run; gated on `ANTHROPIC_API_KEY` +
  `HYDRAPOP_HOME` env vars.
- `src/main/json/` — committed schema JSON, one file per domain.
  Generated by `chatgraph-build-schema <domain>`; sits as a peer of
  `src/main/python/` because the JSON is a build artifact treated as
  another language target, following the Hydra dist convention.
- `config/gremlin/` — Gremlin Server config (yaml + properties + init
  groovy). Copy these into `$GREMLIN_SERVER_HOME/conf/` before launching.
- `docs/` — human-facing documentation (`medical-schema.md`,
  `gremlin-setup.md`). The top-level `README.md` is the entry point.
- `bin/diagnose.py` — one-shot health check of all external
  dependencies.

## Schema: one source of truth

For every domain, the committed Hydra JSON (`src/main/json/<domain>.json`)
is the **single source of truth for the graph schema**. Everything that
needs to know the schema derives from that JSON; nothing maintains a
parallel copy of it by hand.

The chain is one-directional:

1. `domains/<name>/schema_build.py` is a *convenience authoring tool*,
   not a second source of truth and not a runtime dependency. Its only
   job is to produce the JSON ergonomically — expressing the schema as
   readable Python (helpers, shared constants, loops) instead of
   hand-writing a large JSON file. `chatgraph-build-schema <name>`
   regenerates the JSON from it; re-running with no source change
   reproduces the JSON byte-for-byte. Because it is purely an editing
   surface, it is **disposable**: once a schema is finalized and no
   longer being modified, `schema_build.py` could be dropped entirely
   and the committed JSON would stand on its own. The runtime never
   imports it.

   *Future direction:* today `schema_build.py` builds the schema with
   HydraPop's vendored DSL (`hydrapop.dsl.pg`) and serializes it with
   `hydrapop.encode`. These are stopgaps for the pre-0.16 Hydra pin. The
   Hydra-canonical approach is to build the model with Hydra's own
   generated PG DSL and serialize with Hydra's own JSON coder; when
   Hydra 0.16 ships the DSL (`hydra.pg.dsl`, per HydraPop's deletion
   NOTE) and HydraPop is updated, migrate the imports and drop the
   shims. This affects only the authoring surface, not the runtime.
2. `src/main/json/<domain>.json` is the canonical schema artifact, and
   the only thing the runtime loads.
3. The **extractor's schema reference** — the vertex/edge/property table
   appended to the LLM's system prompt — is generated **programmatically**
   from that JSON at `Extractor.__init__`
   (`chat/extractor.py::_allowlists_from_schema` +
   `_format_schema_reference`). It is pure deterministic string-building
   over the decoded JSON; **no LLM is involved in producing it**, so it
   cannot drift from the schema. Likewise the tool-use spec / enums.
4. `docs/<domain>-schema.md` is **for human consumption only**. It is a
   prose walkthrough of the clinical model, loaded by *nothing* in the
   code. If it ever disagrees with the JSON, the JSON wins — treat the
   Markdown as documentation that can lag, never as a spec.

Practical consequence: to change a domain's schema, edit
`schema_build.py`, regenerate the JSON, and (optionally) update the prose
walkthrough to match. Never hand-edit the JSON, and never encode schema
facts (label lists, property types, edge endpoints) anywhere a human has
to keep in sync — derive them from the JSON instead.

## Commit-message conventions

Same as Hydra. The rules are non-negotiable:

- **Short:** 120 characters or less for the whole message.
- **Single line:** subject only, no body paragraphs, no bullet lists.
- **No `Co-Authored-By:` line.** No attribution trailer of any kind.
- **Imperative voice, sentence case.** Examples:
  - `Add Gremlin Server config, serialized graph writer, and Coordinator with barge-in and session resume`
  - `Fix slug normalization in extractor materialization`
  - `Refactor Agent and Extractor to take a domain at construction`

For checkpoint work that isn't ready to be a "real" commit, prefix
with `WIP:`. Example: `WIP: Sketch the bucket-sharing mechanism`. When
ready to finalize, ask the user about squashing: soft-reset WIP
commits and re-commit as **focused topic groups, source changes first,
generated files last**. One commit per logical change — do not collapse
multiple unrelated WIPs into a single combined commit just because
they happened on the same branch.

If a particular change needs more explanation than fits in 120
characters, that's a signal to either (a) split the change into
several commits, or (b) write the explanation as a comment in the
source rather than in the commit log.

## Hard rules

These are non-negotiable.

1. **Never commit secrets.** `.env`, API keys, credentials. The
   gitignore covers `.env`. If you find a secret in the tree, ask the
   user how to scrub it.
2. **Never commit `transcripts/`.** These can contain real
   patient-session content (or any other domain's real interview
   content). Ignored at the repo root.
3. **Never push without explicit user authorization.** Local commits
   are fine; `git push`, `git push --force`, and remote changes
   require the user's go-ahead each time.
4. **Never amend or rewrite published history without authorization.**
   Local-only rewrites (e.g. squashing WIP commits before they're
   pushed) are fine; rewriting commits that have been pushed to a
   remote is not.
5. **Never edit files under `transcripts/`, `.venv/`, `__pycache__/`,
   or any cache directory.** These are either regeneratable or
   per-session output.

## Working with HydraPop

chatgraph depends on a local clone of
[HydraPop](https://github.com/CategoricalData/HydraPop) via the
`HYDRAPOP_HOME` env var. The underlying Hydra runtime
(`hydra-kernel`, `hydra-pg`) is a normal PyPI dependency declared in
`pyproject.toml`; HydraPop itself is the Hydra ↔ TinkerPop bridge plus
the PG DSL helpers (`hydrapop.dsl.pg`, `hydrapop.encode`,
`hydrapop.gremlin_bridge`) and is not yet published. The
`src/main/python/chatgraph/_hydrapop_bootstrap.py` shim prepends
`$HYDRAPOP_HOME/src/main/python` to `sys.path` so `hydrapop.*` imports
resolve.

When chatgraph needs a new Hydra↔TinkerPop helper, the helper goes
into HydraPop, not into chatgraph — chatgraph is the application,
HydraPop is the library. Commits made inside HydraPop on behalf of
chatgraph work should be prefixed `WIP:` per HydraPop's own
conventions, so they can easily be spotted, reviewed, and squashed.

## Session procedure

1. **Identify the working directory.** Most edits happen under
   `src/main/python/chatgraph/`; some land in HydraPop. Read the repo state with
   `git status` and `git log --oneline -10` before deciding scope.
2. **Discuss before doing.** For non-trivial changes, propose the
   approach before writing code.
3. **Commit frequently with short messages.** Use `WIP:` for
   checkpoints. Squash before pushing.
4. **Update documentation alongside code changes.** `README.md`,
   `docs/medical-schema.md`, `docs/gremlin-setup.md`, and this `CLAUDE.md`
   should stay current.
